Teaching with owasp-asi-reference

A guide for instructors who want to use these demos in their security training.

Why use these demos

Traditional security training uses vulnerable apps (DVWA, WebGoat, Juice Shop) to teach attack and defense. owasp-asi-reference does the same for agentic AI — but with a key difference: each demo is a minimal, self-contained Docker stack that runs locally in under a minute. Students see the attack happen, then switch to the defense and watch it fail. That before/after contrast is where the learning sticks.

Recommended teaching arc

The three v1 demos follow a natural pedagogical progression: input → state → action.

1. ASI01 — Agent Goal Hijack (input channel)

Time: 15 minutes · Prerequisites: Docker, Ollama with qwen2.5:7b-instruct

What students learn:

  • How hidden instructions inside a document can hijack an agent's behavior
  • Why retrieved content must never be trusted as instructions
  • The defense pattern: wrap untrusted content in delimiters, strip injection markers, enforce allowlists

Discussion questions:

  • Why doesn't the model "just know" the injected instructions are malicious?
  • What happens when an agent processes email, documents, or web content it didn't generate?
  • How would you detect this in a production system?

Run:

cd ASI01_agent_goal_hijack/attack
docker compose up --abort-on-container-exit
cd ../defense
docker compose up --abort-on-container-exit

2. ASI06 — Memory Poisoning (state channel)

Time: 15 minutes · Prerequisites: Same as ASI01

What students learn:

  • How poisoned memory persists across sessions and affects different users
  • Why memory is a persistent attack surface, not a one-time vulnerability
  • The defense pattern: provenance tagging, trust-scoped recall, quarantine

Discussion questions:

  • How is this different from a prompt injection attack?
  • What real-world systems have persistent memory (RAG, long-term memory)?
  • Why is the victim in this scenario a different person from the attacker?

Run:

cd ASI06_memory_poisoning/attack
docker compose up --abort-on-container-exit
cd ../defense
docker compose up --abort-on-container-exit

3. ASI02 — Tool Misuse (action channel)

Time: 15 minutes · Prerequisites: Same as above

What students learn:

  • How an agent can be induced to use legitimate tools in unsafe ways
  • Why "the prompt says don't do X" is not a real control
  • The defense pattern: runtime policy enforcement at the tool boundary

Discussion questions:

  • What's the difference between ASI01 (tricking the agent) and ASI02 (using tools unsafely)?
  • How do you enforce parameter constraints that the model itself can't be trusted with?
  • What real-world tools have parameter pollution risks?

Run:

cd ASI02_tool_misuse/attack
docker compose up --abort-on-container-exit
cd ../defense
docker compose up --abort-on-container-exit

Classroom setup tips

Minimum viable setup

Each student needs:

  • Docker Desktop (or Docker + Docker Compose)
  • Ollama running locally (ollama serve + ollama pull qwen2.5:7b-instruct)

That's it. No API keys, no cloud accounts, no configuration.

Alternative: shared Ollama instance

If students don't have laptops with enough RAM, run a single Ollama instance on a shared machine and have students connect via OPENAI_BASE_URL.

If Ollama isn't available

Students can use any OpenAI-compatible endpoint with tool-calling support: LM Studio (local), Groq (cloud, free tier), or any local OpenAI-compatible server.

Suggested exercises

Exercise 1: Break the defense

After running the defense demo, have students modify the poisoned_doc.md to use a different injection technique and run the attack against the defense. This teaches that defenses need to be comprehensive.

Exercise 2: Write a new ASI category

Use the CONTRIBUTING.md guide to add a new category. Students copy an existing folder structure, design a new attack scenario, implement the attack and defense, and write the walkthrough.

Exercise 3: Measure defense effectiveness

Have students run the attack demo 10 times and record the success rate. Then run the defense 10 times and record the block rate. Discuss why success rates aren't 100% even with a 7B model.
